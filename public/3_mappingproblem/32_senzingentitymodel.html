Senzing Entity Model

Understanding Senzing’s Data Format

Senzing uses a standardized JSON format for entity resolution. Each record represents either a PERSON or ORGANIZATION entity and contains structured information that Senzing uses to find matches and relationships.

Features vs Payload: The Key Distinction

The most important concept in Senzing mapping is the difference between Features and Payload:

Features (Used for Matching)

Features are attributes that Senzing uses to identify and match entities across records:

  • NAME: Person names (first, last, middle) or organization names
  • ADDRESS: Physical locations (street, city, state, postal code, country)
  • PHONE: Phone numbers with optional type and country
  • EMAIL: Email addresses
  • DATE_OF_BIRTH: Birth dates for persons
  • IDENTIFIERS: SSN, Driver’s License, Passport, National ID, Tax ID, etc.

When Senzing compares records, it analyzes these features to determine if they refer to the same real-world entity.

Payload (Stored But Not Used for Matching)

Payload attributes are operational or contextual data that you want to store with the entity but don’t use for matching:

  • Account balances
  • Customer tiers or status
  • Registration dates
  • Transaction history
  • Internal system IDs
  • Notes or comments

Payload data is returned when you query for entities, but it doesn’t influence the entity resolution process.

Common Mistake: Mapping every field as a feature. Only map fields that should influence entity matching. Everything else should be payload.

Entity Types

Senzing supports two primary entity types:

PERSON Entities

Individual human beings. Typical features include:

{
  "DATA_SOURCE": "CUSTOMERS",
  "RECORD_ID": "1001",
  "RECORD_TYPE": "PERSON",
  "PRIMARY_NAME_LAST": "Smith",
  "PRIMARY_NAME_FIRST": "John",
  "DATE_OF_BIRTH": "1985-03-15",
  "ADDR_LINE1": "123 Main Street",
  "ADDR_CITY": "Springfield",
  "ADDR_STATE": "IL",
  "ADDR_POSTAL_CODE": "62701",
  "PHONE_NUMBER": "555-1234",
  "SSN_NUMBER": "123-45-6789"
}

ORGANIZATION Entities

Companies, government agencies, non-profits, etc. Typical features include:

{
  "DATA_SOURCE": "CUSTOMERS",
  "RECORD_ID": "2001",
  "RECORD_TYPE": "ORGANIZATION",
  "PRIMARY_NAME_ORG": "Acme Corporation",
  "ADDR_LINE1": "456 Business Park Drive",
  "ADDR_CITY": "Chicago",
  "ADDR_STATE": "IL",
  "ADDR_POSTAL_CODE": "60601",
  "PHONE_NUMBER": "555-9876",
  "TAX_ID_NUMBER": "12-3456789"
}

Notice that organizations use NAME_ORG instead of NAME_FIRST/NAME_LAST, and might have TAX_ID_NUMBER instead of SSN_NUMBER.

Required Fields

Every Senzing record must include:

  1. DATA_SOURCE (required) - Identifies where the record came from (e.g., “CUSTOMERS”, “WATCHLIST”)

    • Must be uppercase
    • Must be registered in Senzing configuration before loading data

  2. RECORD_ID (required) - Unique identifier within the data source

    • Must be unique within the DATA_SOURCE
    • Can be any string value

  3. RECORD_TYPE (recommended) - “PERSON” or “ORGANIZATION”

    • Not strictly required, but highly recommended
    • Helps Senzing apply appropriate matching rules

Common Features Reference

Here are the most frequently used Senzing features:

Name Features

  • NAME_FIRST, NAME_LAST, NAME_MIDDLE - For persons
  • NAME_PREFIX, NAME_SUFFIX - Titles and generational suffixes
  • NAME_ORG - For organizations
  • NAME_FULL - When you have a single full name field

Address Features

  • ADDR_LINE1, ADDR_LINE2 - Street address
  • ADDR_CITY - City name
  • ADDR_STATE - State/province code
  • ADDR_POSTAL_CODE - ZIP or postal code
  • ADDR_COUNTRY - Country code

Contact Features

  • PHONE_NUMBER - Phone number
  • EMAIL_ADDRESS - Email address

Demographic Features

  • DATE_OF_BIRTH - Birth date (format: YYYY-MM-DD)
  • GENDER - Gender (M/F/X)
  • CITIZENSHIP - Country of citizenship
  • NATIONALITY - Nationality

Identifier Features

  • SSN_NUMBER - Social Security Number
  • DRIVERS_LICENSE_NUMBER - Driver’s license
  • PASSPORT_NUMBER - Passport
  • NATIONAL_ID_NUMBER - National ID
  • TAX_ID_NUMBER - Tax ID (for organizations)

Don’t memorize these! The AI mapping assistant will guide you through selecting appropriate features based on your source data. This reference is just to give you a mental model.

Flattened JSON Format

Senzing supports two JSON formats: nested (with FEATURES arrays) and flattened (attributes at root level). This workshop uses the flattened format because it’s simpler:

Flattened (what we’ll use):

{
  "DATA_SOURCE": "CUSTOMERS",
  "RECORD_ID": "1001",
  "PRIMARY_NAME_FIRST": "John",
  "PRIMARY_NAME_LAST": "Smith",
  "DATE_OF_BIRTH": "1985-03-15"
}

The PRIMARY_ prefix indicates this is the primary name feature for the entity.

Key Takeaways

  1. Features are used for matching, Payload is stored but not matched
  2. PERSON vs ORGANIZATION entities require different attributes
  3. DATA_SOURCE, RECORD_ID, and RECORD_TYPE are essential fields
  4. Senzing supports dozens of standard features with specific attribute names
  5. The flattened format puts all attributes at the root JSON level

Remember: You don’t need to memorize all feature types. The Mapping Assistant will reference the specifications for you and recommend appropriate features based on your source data.
