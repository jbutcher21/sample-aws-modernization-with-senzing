The Senzing Folder Tour

Understanding Your Senzing Toolkit

The senzing/ folder contains everything you need to work with AI and validate your mappings. Let’s walk through each file to understand what it does and when you’ll use it.

senzing/
├── prompts/
├── reference/
└── tools/

Prompts Folder

prompts/senzing_mapping_assistant.md

What it is: A structured 5-stage prompt that guides AI through the entity resolution mapping process.

Size: ~500 lines

The 5 stages:

  1. INIT - Load reference materials into AI context
  2. INVENTORY - AI analyzes your source data schema
  3. PLANNING - Determine DATA_SOURCE codes and entity types
  4. MAPPING - Make field-by-field mapping decisions
  5. OUTPUTS - Generate mapper code and documentation

When you use it: At the start of Exercise step 3 (mapping), you’ll load this entire prompt into Amazon Q Developer.

How it helps: Instead of asking vague questions like “write me a mapper,” this prompt gives AI a structured workflow to follow. It knows what questions to ask, what decisions need to be made, and what outputs to generate.

Key feature: The prompt includes explicit instructions for the AI to:

  • Ask you questions before making assumptions
  • Explain its reasoning
  • Generate both code AND documentation
  • Follow Senzing best practices

Reference Folder

These files contain the knowledge AI needs to map your data correctly.

reference/senzing_entity_specification.md

What it is: Complete documentation of every Senzing feature and attribute.

Size: 2,100+ lines

What’s inside:

  • All feature types (NAME, ADDRESS, PHONE, EMAIL, etc.)
  • Required and optional attributes for each feature
  • Data format expectations
  • Feature vs Payload distinction
  • Entity types (PERSON vs ORGANIZATION)
  • Canonical attribute names

When you use it: During Stage 1: INIT of the mapping workflow, you’ll load this file into Amazon Q so the AI has complete knowledge of Senzing’s data model.

Why it matters: This is the “brain” that makes AI useful for mapping. Without this loaded, AI would have to guess at Senzing attribute names. With it loaded, AI knows exactly what features exist and how to use them.

Example of what’s in it:

## NAME Features

### PRIMARY_NAME_FIRST
- Description: Given name for a person
- Required: No
- Format: String
- Example: "John"

### PRIMARY_NAME_LAST
- Description: Family name for a person
- Required: No
- Format: String
- Example: "Smith"

[... and 50+ more NAME attributes ...]

reference/senzing_mapping_examples.md

What it is: Real-world examples of common mapping scenarios.

Size: ~400 lines

What’s inside:

  • Complete example mappings for various data types
  • How to handle names (persons vs organizations)
  • Address mapping patterns
  • Identifier handling (SSN, passport, driver’s license)
  • Conditional logic examples
  • Best practices and common patterns

When you use it: Load during Stage 1: INIT alongside the entity specification. AI references these examples when suggesting mappings.

Why it matters: Examples help AI understand mapping patterns and make better suggestions. Instead of just knowing “SSN_NUMBER exists,” AI sees how it’s used in practice.

Example of what’s in it:

# Example: Conditional entity type mapping
if row['customer_type'] == 'Company':
    record['RECORD_TYPE'] = 'ORGANIZATION'
    record['PRIMARY_NAME_ORG'] = row['customer_name']
else:
    record['RECORD_TYPE'] = 'PERSON'
    record['PRIMARY_NAME_FIRST'] = row['first_name']
    record['PRIMARY_NAME_LAST'] = row['last_name']

reference/identifier_crosswalk.json

What it is: Lookup table of standard identifier types and their aliases.

Size: ~100 lines JSON

What’s inside:

{
  "SSN": {
    "senzing_attribute": "SSN_NUMBER",
    "aliases": ["social security number", "ssn", "social_security_num"],
    "description": "US Social Security Number"
  },
  "PASSPORT": {
    "senzing_attribute": "PASSPORT_NUMBER",
    "aliases": ["passport", "passport_num", "passport_number"],
    "description": "Passport identifier"
  }
}

When you use it: Load during Stage 1: INIT so AI can correctly map identifier fields.

Why it matters: Your source data might call identifiers by different names (“SSN”, “social_security_number”, “tax_id”). This crosswalk helps AI recognize these variants and map to the correct Senzing attribute.

reference/usage_type_crosswalk.json

What it is: Lookup table for usage types (how phone numbers, addresses, and emails are used).

Size: ~80 lines JSON

What’s inside:

{
  "PHONE_TYPES": ["MOBILE", "HOME", "WORK", "FAX"],
  "EMAIL_TYPES": ["PERSONAL", "WORK"],
  "ADDRESS_TYPES": ["HOME", "WORK", "MAILING"]
}

When you use it: Load during Stage 1: INIT alongside other references.

Why it matters: If your source data has fields like phone_type or email_type, AI needs to know what valid values Senzing expects.

reference/lint_senzing_json.py

What it is: Python script that validates Senzing JSON syntax and structure.

What it checks:

  • Valid JSON syntax (no parsing errors)
  • Required fields present (DATA_SOURCE, RECORD_ID)
  • Line-by-line validation with error reporting

When you use it: During Exercise step 4 (validation), after running your mapper.

Command:

python3 workshop/senzing/reference/lint_senzing_json.py output.jsonl

Output example:

✓ Line 1: Valid
✓ Line 2: Valid
✗ Line 3: ERROR - Missing required field: DATA_SOURCE
✓ Line 4: Valid
...
Summary: 119 valid, 1 error

Why it matters: Catches basic JSON errors before you try to load data into Senzing. Much faster than discovering errors during the load process.

Tools Folder

These scripts help you analyze data at various stages of the workflow.

tools/sz_schema_generator.py

What it is: Python script that analyzes source data and generates detailed schema documentation.

What it analyzes:

  • Field names
  • Data types
  • Sample values
  • Population rates (how often fields are filled)
  • Cardinality (number of unique values)
  • Data quality notes

When you use it: During Exercise step 2 (schema creation), before you start mapping.

Command:

python3 workshop/senzing/tools/sz_schema_generator.py workspace/customers/customers.csv -o customers_schema.md

Output: A markdown file documenting your source data structure.

Why it matters: The schema file is what you provide to AI during Stage 2: INVENTORY of the mapping workflow. It gives AI complete visibility into your data structure without having to process the entire dataset.

Example output:

## Field: customer_name
- **Type**: string
- **Population**: 100% (120/120 records)
- **Unique Values**: 118
- **Sample Values**:
  - "John Smith"
  - "Acme Corporation"
  - "Maria Garcia"

## Field: date_of_birth
- **Type**: date
- **Population**: 95% (114/120 records)
- **Sample Values**:
  - "1985-03-15"
  - "1972-11-20"

tools/sz_json_analyzer.py

What it is: Python script that analyzes mapped Senzing JSON to show which attributes are recognized and will be used for matching.

What it shows (color-coded):

  • GREEN: Senzing features that will be used for entity matching
  • YELLOW: Payload attributes (stored but not used for matching)
  • RED: Errors (unregistered data sources, unknown attributes)
  • ORANGE: Warnings (sparsely populated fields)

When you use it: During Exercise step 4 (validation), after linting your output.

Command:

python3 workshop/senzing/tools/sz_json_analyzer.py output.jsonl

Output example:

Analyzing: output.jsonl (120 records)

DATA_SOURCE: CUSTOMERS [GREEN - registered]

Features (will be used for matching):
  PRIMARY_NAME_FIRST: 114 records [GREEN]
  PRIMARY_NAME_LAST: 114 records [GREEN]
  PRIMARY_NAME_ORG: 6 records [GREEN]
  DATE_OF_BIRTH: 114 records [GREEN]
  SSN_NUMBER: 45 records [GREEN]
  ADDR_LINE1: 118 records [GREEN]
  ADDR_CITY: 118 records [GREEN]

Payload (stored, not matched):
  CUSTOMER_SINCE_DATE: 120 records [YELLOW]
  ACCOUNT_STATUS: 120 records [YELLOW]
  ACCOUNT_BALANCE: 120 records [YELLOW]

Warnings:
  PASSPORT_NUMBER: 12 records [ORANGE - sparse population]

Why it matters: This tells you EXACTLY what Senzing will do with your mapped data. Green attributes will influence entity resolution. Yellow attributes are just stored. Red means there’s a problem that must be fixed.

How These Files Work Together

Example: Complete mapping workflow

Step 1 - Generate schema:

python3 senzing/tools/sz_schema_generator.py workspace/customers/customers.csv -o schema.md

Step 2 - Load files into Amazon Q (Stage 1: INIT):

  • senzing/prompts/senzing_mapping_assistant.md
  • senzing/reference/senzing_entity_specification.md
  • senzing/reference/senzing_mapping_examples.md
  • senzing/reference/identifier_crosswalk.json
  • senzing/reference/usage_type_crosswalk.json

Step 3 - Provide schema to AI (Stage 2: INVENTORY):

  • Share the schema.md file

Step 4 - Work through Stages 3-5 with AI

  • AI has all the knowledge it needs
  • You provide business context
  • AI generates mapper code

Step 5 - Validate output:

# Run mapper
python3 customers_mapper.py workspace/customers/customers.csv -o output.jsonl

# Lint
python3 senzing/reference/lint_senzing_json.py output.jsonl

# Analyze
python3 senzing/tools/sz_json_analyzer.py output.jsonl

Result: Production-ready mapper with validated output!

Key Takeaways

Reference files (specification, examples, crosswalks):

  • Loaded into AI during Stage 1: INIT
  • Give AI complete knowledge of Senzing
  • Enable accurate mapping suggestions

Validation tools (linter, analyzer):

  • Used after mapping to verify correctness
  • Catch errors before loading into Senzing
  • Show exactly what Senzing will do with your data

Analysis tools (schema generator):

  • Used before mapping to understand source data
  • Generate documentation AI needs for Stage 2: INVENTORY
