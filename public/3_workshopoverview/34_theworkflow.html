The Workflow

The 7-Step Repeatable Process

Both exercises (customer data and watchlist data) follow the same workflow. Learn this pattern once, use it for any dataset.

Step What You Do Tools Time
1. Examine View and understand source data Terminal, file viewers 5 min
2. Schema Generate data structure documentation sz_schema_generator.py 5-10 min
3. Map Work with AI to create mapper Amazon Q + Mapping Assistant 30-40 min
4. Validate Check mapper output Linter + Analyzer 10-15 min
5. Load Put data into Senzing sz_configtool + sz_file_loader 5 min
6. Snapshot Generate resolution analysis sz_snapshot 3-5 min
7. Analyze Explore resolved entities sz_explorer + MCP 10-15 min

Total time per exercise: 60-90 minutes

Step 1: Examine the Source Data

Goal: Understand what you’re working with

What to do:

# For CSV
cd /home/ubuntu/workshop/workspace/customers
head -20 customers.csv

# For JSON
cd /home/ubuntu/workshop/workspace/watchlist
head -3 ftm.jsonl | python3 -m json.tool

What to look for:

  • Field names
  • Data types
  • Sample values
  • Special characteristics (nested data, conditional fields, etc.)
  • Data completeness

Checkpoint: Can you describe your data structure? Do you understand what each field represents?

Step 2: Create the Schema

Goal: Generate documentation of your source data that AI will analyze

Command:

cd /home/ubuntu/workshop
python3 senzing/tools/sz_schema_generator.py workspace/customers/customers.csv -o customers_schema.md

What the tool creates:

  • Field-by-field analysis
  • Data types and population rates
  • Sample values
  • Cardinality (unique value counts)

Review the output:

cat customers_schema.md | head -100

Checkpoint: Do you have a schema file ready to share with AI?

Step 3: Map with AI Assistant

Goal: Use the 5-stage Senzing Mapping Assistant to create your mapper

This is the core of the exercise. You’ll work through 5 stages with Amazon Q Developer.

Stage 1: INIT - Load Reference Materials

What to ask AI:

I'm going to load the Senzing Mapping Assistant prompt and reference files.
Please confirm when you're ready.

Then load these files (copy-paste content into Amazon Q):

  1. workshop/senzing/prompts/senzing_mapping_assistant.md
  2. workshop/senzing/reference/senzing_entity_specification.md
  3. workshop/senzing/reference/senzing_mapping_examples.md
  4. workshop/senzing/reference/identifier_crosswalk.json
  5. workshop/senzing/reference/usage_type_crosswalk.json

AI’s response: Will confirm it’s loaded the materials and is ready for Stage 2.

Your role: Just load the files. AI does the reading.

Stage 2: INVENTORY - Analyze Source Data

What to ask AI:

I have a schema file for my source data. I'm providing it now so you can analyze
the fields and understand what we're working with.

[Paste contents of customers_schema.md]

AI’s response: Will analyze all fields and summarize what it found (field names, types, population rates).

Your role: Provide the schema. Answer any clarifying questions AI has about the data.

AI questions you might get:

  • “I see a field called customer_type. What values does this contain?”
  • “The id_number field - does this contain different types of identifiers?”

Stage 3: PLANNING - Determine Strategy

What AI will ask:

  • “What DATA_SOURCE code should we use?”
  • “Are there multiple entity types in your data (PERSON and ORGANIZATION)?”
  • “Do any fields require conditional logic?”

Your responses (examples):

  • “Use DATA_SOURCE: CUSTOMERS”
  • “Yes, the customer_type field indicates ‘Individual’ or ‘Company’”
  • “The id_type field tells us if id_number is an SSN, Driver’s License, or Passport”

AI’s response: Will confirm the strategy and prepare for field mapping.

Your role: Make business decisions about how your data should be structured. AI suggests, you decide.

Stage 4: MAPPING - Field-by-Field Decisions

This is where the main work happens. For each source field, AI will suggest a mapping and you’ll confirm or correct it.

AI’s approach (for each field):

  1. Suggest if it’s a Feature (for matching) or Payload (just stored)
  2. Recommend the Senzing attribute name
  3. Note any transformations needed

Example conversation:

AI: “For customer_name, I suggest:

  • For PERSON records: Split into NAME_FIRST and NAME_LAST
  • For ORGANIZATION records: Map to NAME_ORG
  • Does this align with your data?”

You: “Yes, that’s correct. The name is already split in the data - we have separate first_name and last_name columns for individuals.”

AI: “Perfect. For date_of_birth:

  • Map to DATE_OF_BIRTH feature
  • Only for PERSON records
  • Format appears to be YYYY-MM-DD already
  • Correct?”

You: “Correct.”

AI: “For registration_date:

  • I notice this might be a customer signup date, not a legal entity registration date
  • Suggest mapping as payload: CUSTOMER_SINCE_DATE
  • Or should this be the Senzing REGISTRATION_DATE feature?”

You: “Good catch! This is indeed customer signup, not legal registration. Use CUSTOMER_SINCE_DATE as payload.”

What to ask AI when you need guidance:

  • “What’s the difference between SSN_NUMBER and NATIONAL_ID_NUMBER?”
  • “Should account_balance be a feature or payload?”
  • “How do I handle a field that has multiple identifier types?”
  • “Show me an example of conditional mapping based on entity type”

Your role:

  • ✅ Confirm AI suggestions when they’re correct
  • ✅ Correct AI when it misunderstands your data
  • ✅ Ask for clarification when you’re unsure
  • ✅ Provide business context AI doesn’t have

AI’s role:

  • ✅ Suggest mappings based on Senzing specifications
  • ✅ Explain why suggestions are appropriate
  • ✅ Show examples from reference materials
  • ✅ Ask clarifying questions before assuming

Stage 5: OUTPUTS - Generate Code and Documentation

What to ask AI:

We've completed the mapping decisions. Please generate:
1. The Python mapper code
2. The mapping specification document
3. A README with usage instructions

AI’s response: Will provide:

  • Mapper code (customers_mapper.py): Production-ready Python script
  • Mapping spec (customers_mapper.md): Documentation of all decisions
  • README: How to run the mapper

Save the files:

# Save each output to the appropriate file
cd /home/ubuntu/workshop
# Copy AI output to files

Your role: Review the generated code. Ask for changes if needed:

  • “Add better error handling for missing values”
  • “Add a progress indicator when processing records”
  • “Include more comments explaining the conditional logic”

Step 4: Validate the Mapping

Goal: Verify your mapper produces correct output

Run the mapper:

python3 customers_mapper.py workspace/customers/customers.csv -o customers_output.jsonl

Lint the output:

python3 senzing/reference/lint_senzing_json.py customers_output.jsonl

Expected result: “✓ All records valid”

Analyze the output:

python3 senzing/tools/sz_json_analyzer.py customers_output.jsonl

Look for:

  • GREEN attributes: Features that will be used for matching (should include NAME, ADDRESS, identifiers)
  • YELLOW attributes: Payload (operational data)
  • RED errors: Must fix these!
  • ORANGE warnings: Review these (might be okay if fields are sparsely populated)

Checkpoint questions:

  • ✅ Did all important source fields get mapped?
  • ✅ Are Features (GREEN) appropriate for entity matching?
  • ✅ Is Payload (YELLOW) appropriate for operational data?
  • ✅ Are there any RED errors that must be fixed?

If you find issues: Go back to AI and ask for corrections.

Step 5: Load Data in Senzing

Goal: Register your data source and load mapped data

Register the data source:

sz_configtool add-datasource --datasource CUSTOMERS

Load the data:

sz_file_loader --file customers_output.jsonl --datasource CUSTOMERS

What happens: Senzing processes each record and resolves entities (finds duplicates and matches).

Checkpoint: Did the load complete successfully? Check for error messages.

Step 6: Take a Snapshot

Goal: Generate entity resolution analysis files

Run snapshot:

sz_snapshot --outputDir snapshot_customers --datasources CUSTOMERS

What’s created: A directory with analysis files showing:

  • How many entities were found
  • Which records matched together
  • Resolution statistics
  • Data quality metrics

Checkpoint: Do you have a snapshot directory with analysis files?

Step 7: Analyze Results

Goal: Explore and understand what Senzing found

Use sz_explorer:

sz_explorer snapshot_customers/

Browse through the interactive interface to see resolved entities.

Use Senzing MCP via Amazon Q:

Ask questions like:

What data sources are configured in Senzing?

Get entity statistics for the CUSTOMERS data source

Search for entities with the name "Smith"

Show me entity with ID 1 and explain why records matched

What you’ll discover:

  • How many unique entities were found from your 120 records
  • Which customer records Senzing determined are the same person/organization
  • WHY Senzing matched them (which features aligned)
  • Insights about data quality

Checkpoint: Do the resolution results make sense? Do matched entities look like actual duplicates?

Your Role vs AI’s Role Throughout the Workflow

AI’s Responsibilities:

  • ✅ Reading and understanding Senzing specifications
  • ✅ Suggesting appropriate feature mappings
  • ✅ Generating mapper code quickly
  • ✅ Creating documentation automatically
  • ✅ Explaining Senzing concepts and best practices

Your Responsibilities:

  • ✅ Understanding YOUR source data
  • ✅ Making business decisions (what should influence matching?)
  • ✅ Providing context AI doesn’t have
  • ✅ Validating AI suggestions against requirements
  • ✅ Correcting AI when it misunderstands
  • ✅ Owning the quality of the final mapper

Working Together Effectively:

When AI suggests something you’re unsure about:

  • Ask for clarification: “Why did you suggest mapping this to REGISTRATION_DATE instead of payload?”
  • Request examples: “Show me an example of how NATIONAL_ID_NUMBER is used”

When AI makes a mistake:

  • Correct it directly: “No, registration_date in my data is customer signup, not legal entity registration. This should be payload.”
  • Provide context: “In our business, account_status doesn’t influence entity matching. This should be payload.”

When you need options:

  • Ask for alternatives: “What are my options for handling this identifier field?”
  • Request comparisons: “What’s the difference between SSN_NUMBER and NATIONAL_ID_NUMBER?”

Think of the workflow as a structured conversation where AI provides expertise on Senzing specifications, and you provide expertise on your data and business requirements. Neither side has all the answers alone.

Common Questions During Mapping

“Should this field be a Feature or Payload?”

  • Feature: If it should influence entity matching (names, addresses, identifiers, demographics)
  • Payload: If it’s operational data you want to store but not match on (balances, dates, status codes)

“AI suggested REGISTRATION_DATE but I’m not sure if that’s right”

  • Ask AI: “What does Senzing use REGISTRATION_DATE for?”
  • AI will explain it’s for legal entity registration dates
  • You decide: “My registration_date is customer signup, not legal registration, so I’ll use payload instead”

“My data has an id_type field that says ‘SSN’, ‘PASSPORT’, or ‘DL’. How do I handle this?”

  • This needs conditional mapping
  • Ask AI: “Show me how to map dynamic identifiers based on a type field”
  • AI will provide conditional logic example

“AI mapped my customer_name field but some records are companies, not people”

  • This needs conditional mapping based on entity type
  • Tell AI: “Records with customer_type='Company' should use NAME_ORG, others should use NAME_FIRST/NAME_LAST”

Key Principles for Success

  1. Start with good schema - The schema file AI analyzes in Stage 2 is critical
  2. Be explicit about business context - AI knows Senzing, you know your data
  3. Validate at every step - Don’t skip the linter and analyzer tools
  4. Ask questions when unsure - AI can explain Senzing concepts
  5. Review generated code - AI writes it fast, but you should understand it
  6. Iterate if needed - Easy to go back and adjust mappings
